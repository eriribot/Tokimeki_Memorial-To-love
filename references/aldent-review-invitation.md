# 第二集模板化改造：变动说明与台式机手测手册

这份手册对应 2026-07-22 的当前工作区。前半部分解释这轮为什么重构、以后怎么加新集；后半部分用于在台式机和真实 SillyTavern 环境里验收第二集。

实际结果请填在同目录的 `aldent-human-review-form.md`。没有看到的证据写“未确认”，不要用生成内容看起来正确来代替世界书扫描证据。

## 手测前先处理这件事

运行时不会读取 `data/lore-books/*.txt`，也不会读取仓库里的 `出包王女 (2).json`。它会直接读取 SillyTavern 中名为 `出包王女` 的真实世界书。

当前 `出包王女 (2).json` 仍是旧内容：order 153 里还有“本幕直接承接”，order 154 仍把最后一小时的谈话写在家中。直接导入这份 JSON，会生成旧版本剧情。

开始手测前，请确认酒馆中 order 152、153、154 的正文已经换成下面三份当前草稿：

- `data/lore-books/tolove-tv-episode-02-act01.txt`
- `data/lore-books/tolove-tv-episode-02-act02.txt`
- `data/lore-books/tolove-tv-episode-02-act03.txt`

如果酒馆里的条目已经是当前内容，只需逐项核对，不要再次导入。本文没有修改本地 JSON，也没有替你写入酒馆世界书。

还有一处已经确认的文字与技术合同差异：第二幕 TXT 写“本幕在两人动身时结束”，代码的场景合同却要求本幕最后至少出现一次 `riverbank`。当前运行时的判定是：最后可以用河边过场或刚到河堤收尾，但不能展开第三幕的心声和警报。如果模型一直停在 `home` 场景，即使文字写了“去河边”，也会被解析器判为缺少 `riverbank`。这轮按要求只写文档，没有替你决定要改 TXT 还是改场景合同。

## 这轮改了什么

### 为什么原来的公共文件会变长

上一版把第一集的触发、幕进度、生成状态、楼层选择、旧存档恢复和正文投影都写进了公共 Store 与快照。每接一集都要增加一组字段和分支，第二集加入后自然会继续膨胀。

这部分专用结构已经拆掉：

- `stores/gameStore.ts` 从 511 行降到 186 行，现在只负责通用行动结算、日期和主线接口装配。
- `save/snapshot.ts` 从 487 行降到 191 行，不再恢复第一集专用字段，也不迁移开发期旧存档。
- `entryReason`、持久化 `actIndex`、`generationId`、重复的正文投影和幕内 AP 结算字段已经删除。
- 主线状态统一为 `mainStory`；当前游标只保存 `eventId + actId + phase + pageIndex`。

这些逻辑不是换个文件继续按集复制。`stores/mainStoryStore.ts`、`GalMainStory/storyPersistence.ts` 和 `GalMainStory/storyArchive.ts` 处理所有分集共用的触发、生成、候选楼层、历史、存读档和完幕结算，内部没有 EP01 或 EP02 分支。

### 现在怎么接一集剧情

`GalMainStory/episodeTemplate.ts` 定义统一接口。一幕只登记以下内容：

- 触发日期和当天第几次行动；
- 剧情世界书名称、order、条目名和根标签；
- 本幕需要的人物世界书；
- 可用场景、演员、立绘和表情范围；
- 最少正文行数、必经场景顺序和保底页。

`GalMainStory/episodes/index.ts` 是唯一生产注册表。以后新增一集，复用现有人物和场景时，只需：

1. 新建这一集的 act 文件；
2. 用这一集的 `index.ts` 组装元数据和前置事件；
3. 在总注册表增加一次登记。

正常情况下不用再改 `gameStore.ts`、`snapshot.ts`、GAL 播放器或生成服务。注册时会拒绝重复集数、重复幕 ID、倒序触发、重复 worldbook order、无效前置集和循环依赖。

当前运行链如下：

```text
玩家行动
  → 通用模板触发器按日期和行动序号选中 eventId + actId
  → 按世界书名、order 和精确条目名读取真实条目
  → 临时武装下一次 World Info 扫描副本
  → 调用酒馆当前预设生成本幕
  → 校验逐页演出格式、素材范围、行数和场景顺序
  → 保存候选楼层与对话侧档
  → GAL 播放器渲染当前采用楼层
```

### 世界书定位

配置端已经取消 UID。运行时先用“世界书名 + order + 精确条目名”找到条目；同一本世界书里只要 order 重复，就会在生成前报错。

条目读出后，代码仍会使用 SillyTavern 实际返回的 UID，在当次 `WORLDINFO_ENTRIES_LOADED` 数据里认出同一份扫描副本。这只是扫描事件内部的匹配手段，不是剧情配置依据。重新导入世界书导致 UID 变化，不应影响剧情定位。

### 存档和楼层

- 游戏快照是 schema v2；对话档也是 schema v2。
- 旧开发存档会直接提示“存档版本无效；开发阶段旧存档不再兼容，请新开档”，不会尝试部分恢复。
- 楼层和消息通过 `eventId + actId + floorId` 对齐，幕序号由模板推导，不再重复保存。
- 重新生成只继承同一集前面各幕的当前采用楼层。新结果先保存为候选，不会自动替换当前版。
- 后续楼层仍引用某个旧楼层时，旧楼层不能删除。
- 生成过程中如果存档、采用版或前文已经变化，迟到的异步结果不会写回。

## 没有改的范围

- 没有同步 `出包王女 (2).json` 的正文。
- 没有制作、修改或接入新的菈菈立绘。

### 5. 记录真实 order 扫描证据

功能正常不等于已经证明 order 扫描成功。AI 可能凭模型先验写出相似内容。

项目已经暴露只读观察器，可以直接记录 SillyTavern 的 `WORLD_INFO_ACTIVATED` 结果。打开开发者工具后，先把 Console 的执行上下文切到游戏 iframe，在触发一幕前执行：

```js
const storyScan = window.toloveWorldbook.observeActivations();
```

生成进入 GAL 或错误页后执行：

```js
storyScan.getLatest()?.entries.map(({ world, order, comment, disable }) => ({ world, order, comment, disable }));
```

保存结果后执行 `storyScan.stop()`。每幕都重新建立观察器，免得把上一次生成记成当前结果。观察器里的条目是本次扫描副本，显示为启用属于正常情况；酒馆世界书编辑页里的原条目仍应保持关闭。

如果当前环境没有这个观察入口，也可以使用 SillyTavern 自带或已安装工具显示的最终 prompt、Prompt Itemization 或世界书激活列表。每幕分别保存证据：

| 幕 | 本次应出现的剧情 order | 本次应出现的人物 order | 不应出现的剧情条目 |
| --- | ---: | --- | --- |
| 第二集第一幕 | 152 | 100、101、102 | 150、151、153、154 |
| 第二集第二幕 | 153 | 100、102 | 150、151、152、154 |
| 第二集第三幕 | 154 | 100、101 | 150、151、152、153 |

每次生成前后，再打开世界书确认这些保存条目仍是关闭状态。代码只临时启用扫描副本，不应把真实条目改成常开。

如果观察器返回空值，也看不到最终 prompt 或激活列表，就在审查表里写“生成行为通过，精确扫描未确认”。不要根据 UID 判断；UID 可以变化。

常见配置错误会直接显示在“传送坐标中断”面板中：

- 找不到指定 order 和名称；
- 同一本书里存在重复 order；
- 条目被手动开启；
- 根标签不匹配或标签内没有正文；
- 当前环境没有 Tavern Helper 生成或 World Info 扫描接口。

先记录完整错误，再修正世界书并点“重试”。“使用保底版”只用来继续走流程。

行动点在触发主线时已经扣除。生成失败不会退还 AP，“重试”和“使用保底版”也不应再扣一次。若返回内容是 JSON、缺少受支持的正文容器、行数不足、场景顺序不全或包含未登记素材，应记为“已读到世界书但 AI 正文解析失败”，不要和 order 查找失败混为一类。

### 6. 检查存档恢复

先完成第二集第一幕，回到 4 月 9 日地图。此时应剩 1 AP，第二幕处于等待状态。

1. 展开地图菜单，进入“保存数据”，存到一个空的手动槽位。
2. 再执行一次行动，确认第二幕开始。完成第二幕后，打开“读取数据”，读取刚才的 4 月 9 日存档。
3. 读档后应回到 4 月 9 日、1 AP；第一幕仍在已读剧情里，第一幕不会重复触发。
4. 再做一次行动，第二幕应正常开始。
5. 完成第二幕，此时应是 4 月 10 日、2 AP。再存一个槽位并读取。
6. 读档后的第一次行动不应触发第三幕；第二次行动才触发第三幕。
7. 打开“AI 原文”，确认已采用的真实 AI 楼层仍能读取，说明主档和对话侧档成对恢复。

GAL 翻页本身不会单独触发自动存档。只翻了几页就刷新，可能回到最近一次实际写盘时的页，通常是本幕第一页；这不是本轮故障。当前 UI 在剧情覆盖层内也不开放手动存档，所以不要把“幕中任意页精确续读”列为通过条件。

如需检查旧档拒绝行为，可以尝试读取一个 schema v1 开发存档。预期是明确报版本无效，不能静默读取、半恢复或破坏当前新档。这个项目不要求兼容旧档。

### 7. 检查已读剧情、候选楼层和 AI 原文
