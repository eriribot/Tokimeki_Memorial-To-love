# ToLove 桌面端兼容问题交接

更新时间：2026-07-14

## 目标

在台式机环境重新验证开发版与发布版，最终保证：

- `pnpm watch` 可用于实时开发；
- `pnpm build` 产物可用于正式发布，不依赖本机开发路径；
- React、React DOM、JSX Runtime 只加载同一版本和同一实例；
- 图片、字体、音乐等静态资源不出现 404；
- 酒馆助手存档脚本能在不同设备访问同一 SillyTavern 时稳定读写；
- `dist` 内不出现 `savesolt`、`save-bridge` 或存档脚本 JSON。

## 当前已观察到的问题

### P0：笔记本 watch 环境出现 React Hook 崩溃，台式机未复现

笔记本控制台错误：

```text
Uncaught TypeError: Cannot read properties of null (reading 'useCallback')
    at react.production.js:486
    at react.mjs:8
    at react.mjs:16
    at App (App.tsx:37)
```

已知现状：

- 台式机原来的 watch 环境没有出现该错误；
- `package.json` 中 `react`、`react-dom` 当前为 `19.2.7`；
- `webpack.config.ts` 的外链 CDN 当前固定为 `19.2.6`；
- React、React DOM、`react-dom/client`、`jsx-runtime`、`jsx-dev-runtime` 已统一到同一 CDN 域名，但版本仍与本地依赖不一致；
- 不排除浏览器模块缓存、酒馆 iframe 缓存或某个依赖再次解析 React，造成两个 React 实例。

这只是高风险线索，尚未证明是唯一根因。台式机排查时必须先记录 Network 中实际加载的全部 React URL，再决定统一到 `19.2.6` 还是 `19.2.7`，不要只根据本地 `node_modules` 判断。

### P0：watch 环境出现图片和字体 404

曾观察到的请求包括：

```text
/dist/artsource/ui/menu/menu_A03.png
/dist/artsource/ui/menu/menu_B03.png
/dist/font/FOT-SeuratPro-M.otf
/dist/font/Arial.ttf
/artsource/ui/title_bg.png
```

已做的处理：

- `界面-实时修改.json` 会先设置：

  ```js
  window.__WEBGAME_ASSET_BASE__ = 'http://localhost:5500/dist/';
  ```

- `utils/assetPath.ts` 统一用该基址解析以 `/` 开头的资源；
- Webpack 构建后会把 `artsource` 和 `font` 复制到 `dist` 公共资源目录；
- 字体改为运行时根据资源基址注入，不再依赖 iframe 当前 URL 猜路径。

台式机仍需验证静态服务器的根目录。服务器必须以项目根目录为根，使下面的 URL 能直接返回 200：

```text
http://localhost:5500/dist/artsource/...
http://localhost:5500/dist/font/FOT-SeuratPro-M.otf
http://localhost:5500/dist/font/Arial.ttf
```

不要混用以下地址后直接比较结果：

```text
localhost:5500
127.0.0.1:5500
127.0.0.1:8000
```

它们是不同 Origin，也可能对应不同静态服务器根目录和缓存。

### P1：开发注入与正式发布不是同一种方式

当前 `界面-实时修改.json` 只服务于开发：

```text
http://localhost:5500/dist/Tokimeki_Memorial-To-love/index.html
```

正式发布不能继续引用 localhost。正式方案应以 `pnpm build` 后的内联 `index.html` 为来源，再生成或替换发布正则内容。不要把 watch 的本机 URL 当成发布方案。

目前发布正则的最终自动替换流程尚未完成。

### P1：酒馆消息 iframe 的初始高度与浏览器原生全屏

问题根因：标题页原先使用 `100vh` 和多处 `vh` 定位。在酒馆助手 watch 注入时，运行期样式没有经过助手源码中的 `min-height: ...vh` 转换，因此 `vh` 取到的是消息 iframe 当前约 225px 的视口高度，标题页会被压成横条。

当前处理：

- 嵌入标题页改为 `width + aspect-ratio` 支撑真实文档高度，让酒馆助手按 `body.scrollHeight` 自动调整 iframe；
- “全屏游玩”不是单纯把消息 iframe 拉满，而是在 SillyTavern 最外层页面创建覆盖载体，并用 React Portal 保留原酒馆助手/存档上下文；
- 随后对最外层文档调用浏览器原生 Fullscreen API，所以地址栏、标签栏和系统任务栏会隐藏；
- 点击“退出全屏”或按 `Esc` 后退出原生全屏并删除顶层载体；
- 浏览器拒绝 Fullscreen API 时回到嵌入界面并显示错误，不把普通页面覆盖冒充为全屏。

已用不含 `allowfullscreen` 的同源嵌套 iframe 模拟酒馆消息楼层验证：最外层 `document.fullscreenElement` 为 `HTML`，覆盖载体尺寸等于完整浏览器视口，`Esc` 后两者均正确清理。

### P1：shujuku 警告与本次前端崩溃不是同一问题

曾出现：

```text
[shujuku_v120] No injection target lorebook set
```

该警告表示 shujuku 没有配置注入目标世界书。它不能解释 React `useCallback` dispatcher 为 null，也不能解释本地图片和字体 404。排查时不要把三类日志混为一个问题。

## 当前存档实现

数据流：

```text
GAL 前端
  → 酒馆助手事件
  → ToLove存档槽.json 中的角色脚本
  → SillyTavern /api/files 接口
  → SillyTavern 主机本地文件
```

当前槽位文件位于 SillyTavern 数据目录：

```text
data/<用户>/user/files/tokimeki-to-love-save-slot-01.json
...
data/<用户>/user/files/tokimeki-to-love-save-slot-08.json
```

已验证：

- 写入、覆盖、枚举、读取、删除；
- 中文数据往返；
- 覆盖时 UUID 不变、revision 递增；
- 无 `crypto.randomUUID` 时使用 Math 兼容 UUID；
- 手机通过局域网连接同一 SillyTavern 用户时读取的是服务端同一批文件；
- 存档脚本 JSON 不依赖 localhost，也不进入 `dist`。

尚未完成：

- 初始化并使用真正的专用 `/save` 目录；
- 将未来 GAL 对话历史加入存档格式；
- 真同层/伪同层以及 shujuku 楼层兼容桥。

## 台式机复测顺序

### 1. 记录环境，不要先升级依赖

记录：

```text
Node 版本
pnpm 版本
浏览器名称与版本
SillyTavern 版本
酒馆助手版本
shujuku 版本
台式机实际启动静态服务器的命令与根目录
```

不要先删除 lockfile、升级 React 或更新所有依赖，否则会失去“台式机原本正常”的对照环境。

### 2. 验证原台式机 watch 基线

1. 启动原有静态服务器与 `pnpm watch`；
2. 打开 DevTools，勾选 Disable cache；
3. 完整刷新酒馆页面，而不只是重新渲染单个楼层；
4. 保存 Console 与 Network 日志；
5. 筛选 `react`，确认每个 URL 的域名、版本和响应状态；
6. 筛选 `artsource`、`font`、`.mp3`，确认没有 404。

### 3. 对比笔记本问题环境

重点比较：

- React URL 是否同时出现 `19.2.6` 与 `19.2.7`；
- 是否同时来自多个 CDN 域名；
- `react-dom/client` 返回内容内部又引用了哪个 React URL；
- 酒馆页面与 iframe 是否命中旧缓存；
- `__WEBGAME_ASSET_BASE__` 的实际值；
- 404 URL 是否少了 `/dist/`；
- localhost 与 127.0.0.1 是否对应同一个静态服务器。

### 4. 验证正式构建

执行：

```text
pnpm build
```

检查：

- `dist/Tokimeki_Memorial-To-love/index.html` 可独立加载；
- 页面不依赖 watch websocket；
- 页面不依赖开发用 localhost 注入；
- `dist` 中不存在 `savesolt`、`save-bridge`、`ToLove存档槽.json`；
- 独立脚本仍位于 `savesolt/导入到酒馆中/ToLove存档槽.json`。

### 5. 验证 SillyTavern 文件存档

1. 导入并启用 `ToLove存档槽.json`；
2. 保存槽位 01；
3. 确认 SillyTavern 数据目录出现对应 JSON；
4. 刷新酒馆页面后读取槽位；
5. 使用局域网手机读取同一槽位；
6. 覆盖槽位并确认 UUID 不变、revision 增加；
7. 删除槽位并确认物理文件被删除。

## 验收标准

- Console 不再出现 React Hook dispatcher/null 错误；
- React 全套运行时只使用一个明确版本；
- 图片、字体、音乐请求全部返回 200；
- watch 与 build 分别按开发和发布边界工作；
- 正式发布不引用 localhost；
- 标题页在消息 iframe 中完整显示；点击“全屏游玩”后进入浏览器原生全屏，`Esc` 可完整恢复酒馆页面；
- 存档刷新后仍存在，电脑和局域网手机可读取同一文件；
- UUID 在不支持 Crypto API 的环境仍可生成；
- `dist` 保持纯前端输出，不含存档桥目录或脚本 JSON。
