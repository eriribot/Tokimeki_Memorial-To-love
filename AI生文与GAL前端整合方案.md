# To LOVE-Ru 酒馆正文与 GAL 前端 MVP

> 更新日期：2026-07-16
>
> 当前状态：已实现本地规则、四幕正文请求、GAL 播放和失败保底；真实酒馆中的 `TavernHelper.generate()` 仍需人工运行审查。

## 目标口径

这个 MVP 只验证一条纵向链路：

```text
校园行动
→ 代码结算行动点与触发原因
→ 按当前幕扫描剧情和人物资料
→ TavernHelper.generate 使用酒馆当前预设生成纯文本正文
→ 前端按自然段分页并识别“角色名：台词”
→ GAL 播放
→ 代码推进幕、日期和完成标记
```

模型只负责正文表达，不负责行动点、好感、日期、事件触发、资源路径和事件完成。

## 角色权威

- User 是玩家主角，界面中称“你”。不能创造女性 User，也不补运行时没有提供的外貌与过去。
- 夕崎梨子是独立的可发展关系角色，进入角色卡、地图、交谈和好感系统。
- 梨子不是 User，不承担第一集浴室初遇、保护菈菈、误告白和婚约的固定剧情位。
- 梨子与春菜、菈菈或 User 的恋爱关系都不能由第一集世界书锁死；后续路线由玩家选择和状态发展。
- 菈菈第一集的核心关系事件发生在 User 身上，与原作的事故喜剧结构保持对应。

## 行动与时间

每天有 2 点行动点，不再提供独立的“推进时间/结束今日”按钮。

```text
4 月 7 日早晨，AP=2
→ 第一次有效行动，AP=1，时段自动到放学后
→ 此时出现“尝试向春菜告白”
```

第一集有两个入口：

```text
主动告白：
  告白行动消耗最后 1 AP
  outcome = interrupted
  entryReason = confession_interrupted

日终兜底：
  第二次普通行动正常结算并把 AP 用到 0
  entryReason = ap_depleted
```

告白被剧情打断不等于 UI 取消：

- 关闭界面、生成失败和重试不能再次扣点。
- 告白入口不增加春菜好感，不登记成功或失败。
- 主线活动期间锁住其他行动和跨日。
- 第一集结束或跳过后由代码进入 4 月 8 日并恢复 2 AP。
- 其他日期 AP 用完后自动进入次日，避免移除时间按钮后软锁。

| 1   | `ep01.act1-falling-star`     | 菈菈宇宙逃亡；User 向春菜告白被事故打断       |

## 为什么不用 JSON

酒馆负责写正文，前端不要求模型兼任场景数据编译器。

生成输出直接采用自然文本：

```text
警报灯一明一灭，飞行器贴着炮火翻向蓝色星球。
菈菈【认真】：才不要回去继续相亲！
白光轰地划过天空，教室窗户一起发颤。
你【紧张】：西连寺，其实我一直想说……
```

正文协议保持纯文本：

- 每个非空自然段是一页 GAL 正文。
- `角色名【情绪】：台词` 识别为带呈现提示的对白，其余行是叙述；旧的 `角色名：台词` 继续兼容。
- 玩家标签固定为“你”，菈菈标签固定为“菈菈”。解析层同时容错 User、当前玩家名、拉拉、Lala、ララ和已登记人物简称。
- 情绪只使用“微笑、担心、开心、认真、慌张、紧张”，前端把它们映射到菈菈现有 `a-f` 表情素材。

模型不输出 JSON、资源路径、`a-f` 编号或状态差值。`GalMainStory/galAssets.ts` 是 `galbox`
窗口、点击提示、姓名牌和立绘文件的权威映射；模型只提供角色、情绪与正文。

第二幕开始后，菈菈在本幕内持续显示。轮到“你”、其他角色或旁白时只停止菈菈口型，不卸载立绘；只有当前说话人是菈菈时才使用她的
`galbox` 姓名牌。

如果酒馆误返回 JSON，前端会显示错误并要求重试，而不会把数据结构展示成正文。

## 世界书扫描

本地资料以纯文本维护：

- `data/lore-books/tolove-tv-episode-01.txt`
- `data/lore-books/lala-satalin-deviluke.txt`
- `data/lore-books/riko-yuzaki.txt`

每次请求执行两种注入：

1. `position: none, should_scan: true`：注入日期、时段、地点和当前幕人物 ID，交给 SillyTavern 原生世界书扫描。
2. `position: in_chat, should_scan: false`：直接注入上述三份本地权威文本，保证关闭的资料也能由游戏节点明确选中。

人物由幕声明，不让 AI 猜谁在场：

- 第一幕：春菜、菈菈、梨子。
- 第二幕：菈菈、沛凯。
- 第三幕：菈菈、沛凯、萨斯丁。
- 第四幕：春菜、菈菈、梨子。

## 酒馆 API 边界

当前路线调用：

```ts
TavernHelper.generate({
  preset_name: 'in_use',
  user_input: 当前幕正文要求,
  injects: [扫描键, 本地剧情与人物资料],
  max_chat_history: 0,
});
```

酒馆公共聊天尾部不直接加入请求；连续性来自当前存档对应对话档中最近的 User prompt 与 Assistant 正文。

这条路线能验证：

- 酒馆当前预设参与生成。
- 酒馆世界书扫描材料参与请求。
- 返回的纯文本正文进入 GAL。

这条路线不能证明：

- 创建了真实 user/assistant 聊天楼层。
- 触发了 `MESSAGE_SENT`、`/trigger await=true`。
- shujuku、ACU 或数据库钩子已经运行。

因此当前接通标签只能是“真实生成 API 候选，待酒馆人工实测”，不能称为真同层或正式数据库流程。

## 失败与恢复

状态分为：

```text
idle → loading → ready
                 ↘ error → retry
                         → fallback
```

- 行动在进入生成前已经由前端原子结算；重试不扣第二次 AP。
- API 缺失、请求失败、空正文、JSON、页数异常或规划文本污染都会进入可见错误态。
- “使用保底版”是玩家明确选择，正文来源标记为 `fallback`，不会冒充酒馆成功。
- 已生成的连续幕、当前幕和当前页进入存档；加载中的幕恢复为 `idle` 并重新请求，不重复结算行动。

## 验证边界

本地自动化需要证明：

- 页面没有推进时间按钮。
- 梨子进入目标卡并能正常获得好感。
- 主动告白与 AP 耗尽都只激活一次第一集。
- 四幕分别请求纯文本，所有请求都没有 `json_schema`。
- 每幕资料和人物扫描键正确注入。
- API 失败可见，保底版可完成。
- 完成后日期为 4 月 8 日、AP 恢复为 2、完成标记唯一。
- 桌面与手机无正文溢出、控制台错误和资源失败。
- 开发包、生产包的最终内联 HTML 均通过 Tavern replacement/entity/syntax 安全检查。

真实酒馆人工审查还需要确认：

- 请求确实使用预期预设，而不是裸 prompt。
- `WORLD_INFO_ACTIVATED` 命中与幕人物一致。
- 生成时没有意外创建宿主楼层或触发 shujuku/database。
- 实际模型返回正文能被自然段协议稳定分页。
